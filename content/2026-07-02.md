# 動機

現在 Unity の Prefab 差分をいい感じに見れるツールを作ろうとしています。それに伴って YAML の解析器を自作しようとしており、まずは `Unity YAML` の仕様を理解しようと試みました。

# 実験

[こちら](https://unity.com/ja/blog/engine-platform/) の記事が非常に有用なので、この記事の流れに沿って手元で Unity を開きつつ実際に確かめるといった具合に進めます。ただし、手元で確かめる必要がなさそうなほどシンプルな部分はスキップします。

下記リポジトリにて実験を行います。

https://github.com/hashiiiii/unity-yaml-playground

## [YAML の構造](https://unity.com/ja/blog/engine-platform/understanding-unitys-serialization-language-yaml#:~:text=%E3%81%8C%E3%81%A7%E3%81%8D%E3%81%BE%E3%81%99%E3%80%82-,YAML%20%E3%81%AE%E6%A7%8B%E9%80%A0,-YAML%20%E3%81%AF%E3%80%81%E3%80%8CYAML)

まずは [こちら](https://github.com/hashiiiii/unity-yaml-playground/blob/main/Assets/Cylinder.prefab) のファイルの先頭から見ていきます。

### ファイルヘッダー

```yml
%YAML 1.1
%TAG !u! tag:unity3d.com,2011:
```

- `%YAML 1.1` — YAML 1.1 に準拠していることを表します。
  - 仕様: https://yaml.org/spec/1.1/current.html#id895631
- `!u!` — `tag:unity3d.com,2011:` のマクロです。

### オブジェクト定義のヘッダー

```yml
--- !u!1 &5661417331689209760
GameObject:
```

各オブジェクトの定義はこの 2 行のヘッダーで始まります。

- `1` — GameObject の document type を表しています。
  - 対応表: https://docs.unity3d.com/ja/current/Manual/ClassIDReference.html
- 1 行目の `!u!1` の `1` と 2 行目の `GameObject` は常に対応しています。つまり、1 行目に `1` と書かれているにも関わらず、2 行目が `MonoBehaviour` になっているということは起こり得ません。
- `&5661417331689209760` — `File ID` と呼ばれています（詳細は後述）。

### 実験に使った Prefab

今回の確認には下記のような Prefab を利用しています。

![](/public/images/blog/2026-07-02/%202026-07-02%2016.30.46.png)

この Cylinder ゲームオブジェクトに付与された ID が `5661417331689209760` だったということです。

Cylinder には `Transform` コンポーネントがアタッチされていますが、YAML 上では次のように表現されています。

```yml
--- !u!4 &2764467885147141891
Transform:
```

UnityEditor 上で `Debug モード` でインスペクターを見てみると、Prefab に書かれている File ID と同じ値が確認できます。

![](/public/images/blog/2026-07-02/%202026-07-02%2016.44.19.png)

### File ID と GUID

インスペクターにもあるように、File ID は `Local Identifier In File` です。つまりこの Cylinder Prefab の中では一意ですが、プロジェクト全体で見ると被っているものが存在し得ます (これも後述します)。

プロジェクト全体で一意な ID としては `GUID` というものが存在します。GUID は `.meta` ファイル内部に記述されています。

https://github.com/hashiiiii/unity-yaml-playground/blob/33eb7782ec3373cf92ae7cba1cb97a1bd19e2a37/Assets/Cylinder.prefab.meta#L2

これが Cylinder.prefab の GUID です。

大体は、インスペクター上で設定された値がそのまま入っていると思います。

- 例: `m_Layer` は UnityEditor 上での `Layer` が `Default` なので `0`
- 例: `m_Name` には、ゲームオブジェクトの名前 `Cylinder` が入っています

また、先ほど紹介した `FileID` という文字がフィールドに多数見られます。こちらは次以降の項目で詳しく見ていきますが、とりあえず分かりやすい箇所だけ先に触れておきます。

#### `m_Component`

このゲームオブジェクトにアタッチされているコンポーネントがリストされています。

```yml
  m_Component:
  - component: {fileID: 2764467885147141891}
```

`- component: {fileID: 2764467885147141891}` に指定されている値は、Transform コンポーネントのヘッダーに記述されている FileID です。

```yml
--- !u!4 &2764467885147141891
Transform:
  m_ObjectHideFlags: 0
  m_CorrespondingSourceObject: {fileID: 0}
  m_PrefabInstance: {fileID: 0}
  m_PrefabAsset: {fileID: 0}
  m_GameObject: {fileID: 5661417331689209760}
  serializedVersion: 2
  m_LocalRotation: {x: 0, y: 0, z: 0, w: 1}
  m_LocalPosition: {x: 0.64596, y: 0.8118, z: -1.57}
  m_LocalScale: {x: 1, y: 1, z: 1}
  m_ConstrainProportionsScale: 0
  m_Children: []
  m_Father: {fileID: 0}
  m_LocalEulerAnglesHint: {x: 0, y: 0, z: 0}
```

このように、FileID を利用して参照関係を構築しています。

### 登場する File ID の整理

| File ID | 指すもの |
| --- | --- |
| `6260946755249466096` | Plane オブジェクトの Transform |
| `3986811621118340026` | Cube オブジェクトの Transform |
| `4266200641933383501` | Plane.prefab 内の Prefab 要素（Cylinder.prefab） |
| `2119447219891347534` | 上記 Cylinder の Transform 情報のみを strip したプレースホルダー (ヒエラルキー上には存在しない) |
| `6720931628801875981` | Plane.prefab 内の Prefab 要素（Cylinder Variant.prefab） |
| `4021899057343478380` | 上記 Cylinder Variant の Transform 情報のみを strip したプレースホルダー (ヒエラルキー上には存在しない) |

### Transform の親子関係

まずは `6260946755249466096` と `3986811621118340026` から。それぞれの Transform には次のようなフィールドが存在します。

```yml
// ---- 6260946755249466096 ----
  m_Children:
  - {fileID: 3986811621118340026}
  - {fileID: 8337881871497287944}
  - {fileID: 1299338568807264467}
  - {fileID: 2119447219891347534}
  m_Father: {fileID: 0}
// ---- 3986811621118340026 ----
  m_Children: []
  m_Father: {fileID: 6260946755249466096}
```

これはそれぞれの Transform が親子関係を成していることを表現しています。Plane オブジェクトが親で、Cube オブジェクトが子です。

### PrefabInstance

次に `4266200641933383501`, `6720931628801875981` です。ここでは `PrefabInstance` が指定されています。

PrefabInstance は、ファイル内に Prefab がネストして配置された際に利用されます。今回で言うと、Plane.prefab の内部に Cylinder, Cylinder Variant に対して利用されています。

- ヒエラルキー上の親子関係: `m_TransformParent: {fileID: 6260946755249466096}` のような形で指定
- `m_Modification`: オリジナルの Prefab のフィールドに対する参照と、override 側の値が記載される

また `target: {fileID: 2764467885147141891, guid: 05ba59bdbdf954600a21005e3b7bf963, type: 3}` に書かれているように、オリジナルの Prefab に対して fileID と guid を用いて解決をしています。

もう少し細かい話にはなるが、`type: 3` というのは `Library` フォルダ配下のデータを参照するということを意味します。Library フォルダにはアセットのローカルキャッシュがあるので、そこから fileID, GUID を利用して参照するということです。
